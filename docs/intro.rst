**OpenBadgesLib**

Introduction
============

OpenBadgesLib is a Python library for signing and verifying `OpenBadges`_
digital credentials. An OpenBadge embeds a cryptographically signed proof
directly inside a badge image (:term:`SVG` or :term:`PNG`), allowing anyone
to verify the badge's authenticity and the identity of its recipient.

.. _OpenBadges: https://www.imsglobal.org/activity/digital-badges

The library supports **two generations** of the IMS Global OpenBadges specification:

* **OpenBadges 2.0** — :term:`JWS` compact serialisation; SHA-256 hashed
  recipient identity; expiration and revocation checking
* **OpenBadges 3.0** — :term:`JWT-VC` (W3C Verifiable Credentials data model);
  ``OpenBadgeCredential`` with structured issuer and achievement metadata

Both specifications use :term:`RSA` 2048-bit (RS256) and :term:`ECC` NIST P-256
(ES256) keys. Both embed the signed token inside :term:`SVG` and :term:`PNG`
badge images using the same baking format.


Package contents
----------------

The package contains:

* **openbadgeslib** — the core Python library; exposes both OB2 and OB3
* **openbadgeslib.ob2** — OpenBadges 2.0 subpackage (JWS); also
  accessible via the top-level compatibility shims ``badge``, ``signer``,
  ``verifier``
* **openbadgeslib.ob3** — OpenBadges 3.0 subpackage (JWT-VC)
* **config.ini** — configuration file (generated by ``openbadges-init``)
* **CLI tools** — five command-line wrappers around the library

  * ``openbadges-init`` — create an initial directory layout and sample config
  * ``openbadges-keygenerator`` — generate RSA or ECC key pairs
  * ``openbadges-signer`` — sign a badge image for a given recipient
  * ``openbadges-verifier`` — verify the assertion embedded in a signed badge
  * ``openbadges-publish`` — publish badge metadata to a web server


Python version
--------------

The library requires **Python >= 3.10**. It has been tested on Python 3.10,
3.11, 3.12, and 3.14.


How signing works
-----------------

When a badge is signed, the library:

1. Builds a JSON :term:`Assertion` payload containing the recipient's hashed
   email, badge metadata URLs, and issuance timestamp.
2. Signs the payload using :term:`JWS` compact serialisation with the issuer's
   private key (RS256 or ES256).
3. Embeds the resulting JWS token inside the badge image:

   * **SVG** — as an ``<openbadges:assertion verify="…"/>`` XML element
   * **PNG** — as an ``iTXt`` chunk with keyword ``openbadges``


How verification works
----------------------

Verification performs the following checks in order:

1. Extract the JWS token from the image file.
2. Download the issuer's public key from the URL embedded in the token.
3. Verify the JWS cryptographic signature.
4. Check the badge has not been revoked (via the issuer's revocation list).
5. Check the badge has not expired.
6. Verify the recipient identity by comparing the SHA-256 hash of the supplied
   email address against the hashed identity in the assertion.


v1.0.1 changes
--------------

Version 1.0.1 is a major compatibility, modernisation, and feature release:

* **OpenBadges 3.0 support** — new ``openbadgeslib.ob3`` subpackage
  implementing the W3C Verifiable Credentials data model:
  ``OpenBadgeCredential``, ``Issuer``, ``Achievement``; ``OB3Signer`` for
  JWT-VC issuance and SVG/PNG baking; ``OB3Verifier`` for verification and
  token extraction.
* **OpenBadges 2.0 subpackage** — OB2 implementation moved to
  ``openbadgeslib.ob2``; top-level ``badge``, ``signer``, and ``verifier``
  modules kept as backward-compatible shims.
* **CLI ``--ob-version`` flag** — all four CLI tools accept ``-V {2,3}`` to
  select the specification version. ``openbadges-verifier`` also gains
  ``-k / --pubkey FILE`` for supplying a PEM key directly in OB3 mode.
* **Python 3.10+ support** — removed ``distutils`` (dropped in 3.12);
  packaging migrated to ``pyproject.toml``.
* **pycryptodome** replaces the abandoned ``pycrypto`` package.
* **PyJWT** replaces the bundled custom JWS engine (``3dparty/jws/``
  removed). The ``cryptography`` package handles all RSA and ECC operations.
* **TLS security** — ``download_file`` now uses the system default TLS context
  instead of the deprecated ``PROTOCOL_TLSv1`` / ``ssl.CERT_NONE`` settings.
* **Copyright** — year range updated to 2014-2026 across all source files.
* **Bug fixes** — verifier logic, ECC private-key detection, PNG chunk tag
  handling, and ``hash_email`` str/bytes type coercion.
* **Test suite** — 203 tests, 89 % line coverage.
